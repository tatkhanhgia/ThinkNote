# Brainstorm Report: Authentication and Account Management for ThinkNote

**Date:** 2026-03-22
**Status:** Complete
**Scope:** Adding auth, email verification, user profiles, admin management to ThinkNote

---

## 1. Problem Statement

ThinkNote is a personal knowledge base with no auth, no database, no user management. Goal:
- Username/password login with email verification
- PostgreSQL database integration
- User profile management (name, email, avatar)
- Single admin role with user management panel
- Protection against public registration abuse

**Key constraint:** Personal project. Favor simplicity over enterprise patterns.

---

## 2. Auth Library: Better Auth (RECOMMENDED)

Evaluated: Better Auth, Auth.js v5, Custom JWT, Lucia Auth

**Why Better Auth wins:**
- Auth.js/NextAuth team joined Better Auth Sep 2025; recommended for new projects
- Plugin architecture: admin, email verification, rate limiting are official plugins
- Built-in admin plugin: user CRUD, role assignment, ban/unban, impersonation
- Built-in email verification: requireEmailVerification, sendOnSignUp
- Database sessions by default; first-class Prisma adapter
- next-intl has official starter templates for Better Auth composition
- CLI auto-generates database schema

**Why NOT Auth.js v5:** Maintenance-mode only. No built-in admin/email/rate-limiting.
**Why NOT Custom JWT:** Reimplements solved problems. Violates KISS/YAGNI.
**Why NOT Lucia:** Deprecated early 2025. Maintainer recommends Better Auth.
---

## 3. ORM: Prisma (RECOMMENDED)

Evaluated: Prisma, Drizzle ORM, Raw pg

**Why Prisma wins:**
- Schema-first .prisma files: easiest mental model
- Auto-migrations via prisma migrate dev
- Prisma Studio: free visual database browser
- Better Auth CLI auto-adds auth tables to schema.prisma
- Prisma 7 (late 2025): pure TypeScript, no Rust engine
- Most Better Auth tutorials use Prisma

**Why NOT Drizzle:** Edge optimization irrelevant for personal project. Steeper learning curve.
**Why NOT Raw pg:** No type safety, no migrations, Better Auth expects an ORM.

---

## 4. Email: Resend (RECOMMENDED)

**Why Resend:** Free 3,000 emails/month. Clean SDK. Better Auth docs use it as primary example.
**Why NOT Nodemailer/SMTP:** More config, Gmail spam risk, less reliable delivery.

---

## 5. Admin: /[locale]/admin/* routes + Better Auth admin plugin

- Follows existing [locale] pattern. Reuses layout, middleware, i18n.
- Better Auth admin plugin: listUsers, createUser, banUser, setRole, removeUser, revokeSession
- Admin guard in layout.tsx checks role server-side

---

## 6. Sessions: Database (Better Auth default)

- Required for instant session revocation (admin ban/unban)
- JWT cannot revoke without blocklist; ban would not take effect until expiry

---

## 7. Security

**Better Auth handles:** bcrypt hashing, CSRF, rate limiting, HttpOnly/Secure/SameSite cookies
**You handle:** Middleware route protection, env vars in .env.local, HTTPS, DB SSL, admin double-check
---

## 8. Architecture Integration

### Middleware Composition
1. Better Auth cookie check (no DB call) - redirect unauth from /admin to /login
2. next-intl locale routing (existing behavior)
3. Pass through for public routes

### New Files
- src/lib/auth.ts, src/lib/auth-client.ts
- src/app/api/auth/[...all]/route.ts
- src/app/[locale]/login, register, verify-email, profile pages
- src/app/[locale]/admin/ layout + page + users/page
- src/components/ui/AuthButton.tsx, UserAvatar.tsx
- prisma/schema.prisma, .env.local

### Unchanged
- All existing public pages, markdown loading, API routes, i18n, Tailwind, layout

### DB Schema (auto-generated by Better Auth CLI)
- User: id, name, email, emailVerified, image, role, banned, banReason, timestamps
- Session: id, userId, token, expiresAt, ipAddress, userAgent
- Account: id, userId, accountId, providerId, password, tokens
- Verification: id, identifier, value, expiresAt, timestamps

---

## 9. Implementation Phases

1. Database setup: Prisma + PostgreSQL
2. Better Auth core: server/client config + API route
3. Middleware composition: next-intl + auth
4. Auth pages: login, register, email verification
5. Email verification: Resend integration
6. User profile page
7. Admin panel: layout, dashboard, user management
8. Header integration: auth button/avatar
9. Protect sensitive routes
---

## 10. Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| Middleware composition breaks i18n | Medium | next-intl v4 documented patterns |
| Better Auth breaking changes | Low | Pin version |
| PostgreSQL setup on Windows | Low | Docker/WSL/native |
| Email delivery failures | Low | Resend reliability + retry |
| Avatar upload storage | Low | URL-based first, upload later (YAGNI) |

---

## 11. Success Criteria

- [ ] Register with email/password, requires email verification
- [ ] Login, view/edit profile (name, email, avatar URL)
- [ ] Admin: view users, ban/unban, change roles
- [ ] Protected routes redirect unauthenticated users
- [ ] Existing public pages unaffected
- [ ] i18n works on all new pages
- [ ] No secrets in git

---

## 12. Final Summary

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auth | **Better Auth** | Plugin ecosystem; endorsed successor of Auth.js + Lucia |
| ORM | **Prisma** | Best DX; auto-migrations; CLI integration |
| Email | **Resend** | Free tier; cleanest API; Better Auth primary example |
| Sessions | **Database** | Admin session revocation; Better Auth default |
| Admin | **/[locale]/admin** | Reuses existing architecture |
| Hashing | **bcrypt** | Industry standard; zero config |

**New deps:** ~3 (better-auth, @prisma/client + prisma, resend)
**Effort:** 3-5 focused sessions
**Disruption:** Minimal. Only middleware.ts modified; everything else additive.
